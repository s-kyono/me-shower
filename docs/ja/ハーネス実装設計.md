# Plan Decision 1

- Harnessをどこまで実装するかを決める。

## Codex Skill＋永続Plan State

**概要:**

再利用可能なCodex Skillとして実装する。
意思決定、submit状態、ADR候補、Design Lockを構造化ファイルへ保存する。

**メリット:**

- 一問一答を再利用できる
- セッションをまたいで再開できる
- submit前後を明示的に区別できる
- ADRとPlanの生成を標準化できる
- 4〜8の自律実行へ自然につながる

**デメリット:**

- Promptだけより実装量が増える
- Skillとschemaの設計が必要
- Codexの挙動を完全には拘束できない

**適している条件:**

今後の全Epicで実際に使う開発Harnessにしたい

### 運用構想

**現在の確定候補:**

```text
方式:
Codex Skill＋永続Plan State

操作:
会話形式

入力:
- 案1
- 案2
- 案3
- 質問
- 自由記述
- 再提案
- submit

状態管理:
構造化ファイルへ永続化

対象:
- 意思決定
- 未決事項
- 人間のsubmit状態
- ADR候補
- 確定ADR
- Plan
- Design Lock
- Review結果
- Release Gate結果
```

**受け入れる弱点**

- SkillだけではAIの挙動を完全には強制できない
- schemaと実際のMarkdownがdriftする可能性がある
- 状態更新の整合性チェックが必要
- Codexのセッションごとの差を吸収する設計が必要

**未決事項**

次のPlan Decisionで決める必要がある。

1. Plan Stateの正本はYAMLかJSONか
2. ADR候補をいつ正式ADRへ昇格するか
3. 一問一答の質問生成を固定Question Setにするか動的生成にするか
4. submitを文字列だけで扱うか、状態遷移コマンドとして扱うか
5. Skillを単一Skillにするか、Plan / Executeで分割するか
6. 既存の.codex構成へどう統合するか
