# ハーネス全体像

## ハーネス仕組み

```sh
Interfaces
  Workflowを進行する

Skills
  単一責務を実行する

Schemas
  State構造を保証する

Templates
  長期成果物を残す

Shared
  共通ルールと言語を提供する

1. プラン -> 2.レビュープラン -> 3. 実装方針/実装設計 -> 4.実装 -> 5.レビュー -> 6. 修正/レビューをループ -> 7.gitにてリモートブランチにadd ~ push 8. PR発行
```


```sh
Plan Phase
├── 1. プラン
│   └── 要件・目的・Scope・Non-goalsを決める
│
├── 2. レビュープラン
│   └── 抜け・矛盾・過剰設計・境界違反を突く
│
├── 3. 実装方針 / 実装設計
│   └── 型・責務・データフロー・永続化・テストへ具体化
│
└── ADR確定 + Plan Submit

Execution Phase
├── 4. 実装
├── 5. レビュー
├── 6. 修正 / レビューループ
├── 7. add / commit / push
└── 8. PR発行
```

## Plan

```sh
Plan Phase
├── AIが意思決定ポイントを一問ずつ提示
├── 案1 / 案2 / 案3
├── AI推奨と理由
├── 質問 / 自由記述 / 再提案
├── 人間のsubmit
├── 決定をPlan Stateへ蓄積
├── 重要な「なぜ」をADRへ記録
└── 最後にDesign Lock

Execution Phase
├── 実装
├── 客観レビュー
├── 修正 / 再レビュー
├── Final Release Gate
├── add / commit / push
└── PR発行
```

## 仕組みの不変条件

```text
AI proposal != decision
Human submit == decision
ADR == durable rationale
Plan == current execution contract
Design Lock == autonomous execution boundary
Implementation != accepted implementation
Test passed != release accepted
```

**人間が握るもの:**

- 目的
- Scope
- Non-goals
- 設計上の選択
- トレードオフ
- ADR
- Design Lock

**AIが握るもの:**

- 選択肢の提案
- 比較材料
- 推奨案
- 抜け漏れの追及
- Design Lock後の実装
- レビュー
- 修正
- Git公開
