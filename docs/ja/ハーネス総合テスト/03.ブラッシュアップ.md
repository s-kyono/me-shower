# Execute State PatchとExecution State Schemaの整合性修正

総合テストで、Development HarnessのExecute Skillsが返すState Patchと、`execution-state.schema.yaml`の定義が一致していないことが判明しました。

実在するファイルと実際のPatch契約を調査し、Execute Workflow全体でState PatchがSchema検証を通過するよう修正してください。

## 重要な前提

Git公開操作は別タスクでRepository Publish Agentへ分離しています。

そのため、今回の修正では次を対象外としてください。

* `publish-branch`
* `create-pull-request`
* `/git/staged_files`
* `/git/remote`
* `/git/remote_head`
* `/pull_request/created`
* `/pull_request/base`
* `/pull_request/head`
* `/pull_request/draft`
* commit、push、Pull Request作成に関するState

存在しないGit関連SkillやStateを新規作成しないでください。

別タスクによるRepository Publish Agent分離後の状態を前提として、Development Harness内部のExecute Stateだけを整合させてください。

## 問題

`execution-state.schema.yaml`は`additionalProperties: false`です。

一方で、一部のExecute SkillsはSchemaに存在しないPathへのState Patchを返す、または許可しています。

代表的な不整合:

```text
Skill Patch Path                      Schema
/review/latest_reference             /review/latest_artifact
/release_gate/reference              /release_gate/artifact
/execution/context_summary           未定義
/execution/allowed_changes           未定義
/execution/implementation_summary    未定義
```

InterfaceがState Patch適用後にSchemaを厳密検証すると、正常なSkill Resultでも拒否される可能性があります。

## 目的

次を成立させてください。

1. すべてのExecute Skillが返すPatch PathがSchemaに存在する
2. 同じ意味を持つField名を一つに統一する
3. Skill、Schema、Workflow、Template、Interfaceで名称が一致する
4. Stateへ保存する情報と、Artifactへ保存する情報を区別する
5. Stateを不要なContext本文や巨大な成果物で肥大化させない
6. State Patch適用後のStateが必ずSchema検証を通過する
7. Schemaに定義されているが誰も使わないFieldを残さない
8. Skillが使用するがSchemaに存在しないFieldを残さない

## 修正対象

少なくとも次を調査してください。

* `.codex/harness/development/interfaces/execute/SKILL.md`
* `.codex/harness/development/interfaces/execute/workflow.yaml`
* `.codex/harness/development/schemas/execution-state.schema.yaml`
* `.codex/harness/development/shared/state-patch.schema.yaml`
* `.codex/harness/development/shared/skill-result.schema.yaml`
* `.codex/harness/development/skills/execute/inspect-execution-context/SKILL.md`
* `.codex/harness/development/skills/execute/implement/SKILL.md`
* `.codex/harness/development/skills/execute/review-implementation/SKILL.md`
* `.codex/harness/development/skills/execute/apply-scope-fix/SKILL.md`
* `.codex/harness/development/skills/execute/run-release-gate/SKILL.md`
* `.codex/harness/development/templates/REVIEW.md`
* `.codex/harness/development/templates/RELEASE_GATE.md`
* Execute StateまたはPatch Pathを参照しているその他の実在ファイル

ファイルが存在しない場合は、存在するものとして扱わず、調査結果へ明記してください。

## 命名方針

同一概念に複数の名称を使用しないでください。

原則として、既存Schemaにすでに定義されている次の名称を優先してください。

```text
/review/latest_artifact
/release_gate/artifact
```

したがって、Skill側で次を使用している場合は修正してください。

```text
/review/latest_reference
  → /review/latest_artifact

/release_gate/reference
  → /release_gate/artifact
```

ただし、既存利用状況を調査し、Schema側を変更する方が合理的な場合は理由を示してください。

最終的には名称を一つへ統一し、互換目的の重複Fieldを残さないでください。

## Execution Context

次のFieldについて、Stateへ保存すべきかを判断してください。

```text
/execution/context_summary
/execution/allowed_changes
/execution/implementation_summary
```

判断基準:

### Stateへ保存するもの

* Workflow再開に必要
* 次のSkillが参照する
* 状態遷移や検証に必要
* 小さく構造化されている
* Revision管理が必要

### ArtifactまたはSkill Outputだけに置くもの

* 長文の説明
* Repository Contextの複製
* Design Lockから再取得できる情報
* Diffや実装内容の全文
* 一時的な診断情報
* 再開に不要な要約

Stateへ保存する場合は、Schemaへ明示的に追加してください。

Stateへ保存しない場合は、SkillのState Patch allowlistから削除し、`output`またはArtifact Referenceとして返してください。

## 推奨するState構造

既存設計との整合性を確認したうえで、次のような責務分離を検討してください。

```yaml
execution:
  current_skill: string | null

  context:
    artifact: string | null
    design_lock_revision: integer
    repository_revision: string | null

  implementation:
    artifact: string | null
    revision: integer
    changed_files: []
    diff_hash: string | null

  fix_cycle:
    attempt_count: integer
    maximum_attempts: 5

review:
  latest_result: accepted | changes_required | blocked | null
  latest_artifact: string | null
  implementation_revision: integer | null
  reviewed_diff_hash: string | null

release_gate:
  result: passed | failed | blocked | null
  artifact: string | null
  implementation_revision: integer | null
  checked_diff_hash: string | null
  secret_scan: passed | failed | blocked | null
  privacy_scan: passed | failed | blocked | null
  raw_source_scan: passed | failed | blocked | null
```

これは必須形ではありません。

ただし、次は識別できる必要があります。

* Context取得対象
* Implementation Revision
* 変更対象ファイル
* Review対象Revision
* Review対象diff
* Release Gate対象Revision
* Release Gate対象diff
* 各Artifactへの参照

長文本文をExecution Stateへ直接保存しないでください。

## SkillごとのPatch Allowlist

各Execute Skillについて、変更可能なPathを明確にしてください。

例:

### inspect-execution-context

変更可能:

```text
/execution/current_skill
/execution/context/*
/state
```

変更禁止:

```text
/review/*
/release_gate/*
```

### implement

変更可能:

```text
/execution/current_skill
/execution/implementation/*
/execution/changed_files
/state
```

変更禁止:

```text
/review/*
/release_gate/*
```

### review-implementation

変更可能:

```text
/review/*
/state
```

実装内容やRepositoryを変更してはならない。

### apply-scope-fix

変更可能:

```text
/execution/implementation/*
/execution/changed_files
/state
```

Review Resultを直接acceptedへ変更してはならない。

### run-release-gate

変更可能:

```text
/release_gate/*
/state
```

実装内容やReview Resultを変更してはならない。

実際のState構造に合わせてPathを調整してください。

## State Patch検証

InterfaceがState Patchを適用する前に、次を検証する契約を維持してください。

1. Patch targetが`execution-state`
2. expected revisionが現在Revisionと一致する
3. expected workflow stateが現在Stateと一致する
4. Skillが変更を許可されたPathだけを操作している
5. Root object全体を置換していない
6. RevisionをSkillが直接変更していない
7. Patch適用後に`execution-state.schema.yaml`を通過する
8. Postconditionを満たす
9. requested transitionがWorkflow上有効
10. 検証失敗時はStateを一切変更しない

State更新はatomicに行ってください。

## Workflow Stateの整合性

`workflow.yaml`で使用するState名と、`execution-state.schema.yaml`の`state` enumを完全一致させてください。

次を確認してください。

* Workflowに存在してSchemaにないState
* Schemaに存在してWorkflowにないState
* 同じ意味の別名称
* 到達不能State
* 遷移元が存在しないState
* Terminal Stateなのに遷移が存在するState

Git公開操作をRepository Publish Agentへ分離した後のWorkflowを基準にしてください。

## Artifact Reference

ReviewとRelease Gateの成果物本文はExecution Stateへ直接格納せず、Artifact Referenceを保存してください。

```text
review.latest_artifact
release_gate.artifact
```

Artifact Referenceには、少なくとも次を識別できるようにしてください。

* Artifact PathまたはID
* Artifact Revision
* 対象Implementation Revision
* 対象diff hash

既存Schemaで別構造にする場合は、理由を示してください。

## Scope外

今回は次を修正しないでください。

* Repository Publish Agent
* stage
* commit
* push
* Pull Request作成
* conflict resolution
* merge
* cherry-pick
* tag
* release
* hotfix
* Plan State Schema
* Plan Workflow
* Artifact Writerそのものの実装
* Human Submit処理
* ADR確定処理

ただし、今回のState整合性修正に必要なArtifact Reference Fieldの追加は許可します。

## Validation

修正後、少なくとも次を実行してください。

### 1. Patch Path抽出

Execute Skills内から、State Patchで使用可能としているPathを抽出してください。

次のような文字列を検索してください。

```text
state_patch
allowed_patch
allowed paths
/review/
/release_gate/
/execution/
```

### 2. Schema照合

抽出したすべてのPathについて、`execution-state.schema.yaml`に対応するFieldが存在することを確認してください。

### 3. 未使用Schema Field

Schema上のFieldが、Interface、Workflow、Skillのいずれからも参照されていない場合は報告してください。

不要なFieldは削除し、将来用途として残す場合は理由を明記してください。

### 4. Workflow照合

Workflow StateとSchema enumが完全一致することを確認してください。

### 5. Patch適用テスト

最低限、次のSkill Resultを想定したPatch適用テストを追加または実行してください。

* inspect-execution-context completed
* implement completed
* review-implementation accepted
* review-implementation changes_required
* apply-scope-fix completed
* run-release-gate passed
* run-release-gate failed
* blocked result
* expected revision不一致
* Skill allowlist外Path
* Schemaに存在しないPath
* Root replacement
* Revision直接変更

正常系ではPatch適用後のStateがSchema検証を通過することを確認してください。

異常系ではStateが変更されないことを確認してください。

## 実施手順

1. 実在する対象ファイルを調査する
2. SkillごとのPatch Path一覧を作成する
3. Schemaとの差分を整理する
4. 修正プランを提示する
5. プラン承認を待たず、そのまま修正する
6. Validationを実行する
7. 残存不整合を再検索する
