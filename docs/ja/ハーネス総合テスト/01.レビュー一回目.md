  ### 3. 成果物を永続化する主体が存在しない

  各Skillは以下の成果物内容またはwrite requestを生成します。

  しかしSkillはStateや成果物を直接変更しない契約です。Interfaceも原則としてStateだけを変更すると定義されて

  例:

  Execute Handoffが要求する「accepted PLAN.md」「accepted DESIGN_LOCK.md」が実ファイルとして成立する保証が

  ## 今直さないと致命的

  ### 1. Design Lock逸脱時のPlan復帰が閉じていない

  Executeのblocked stateにはreturn_to_planがありますが、遷移先はabortedだけです。.codex/harness/
  development/interfaces/execute/workflow.yaml:80

  不足しているもの:

  - Design Lockをinvalidatedにする処理
  - 影響を受けたPlan revisionの記録
  - Plan Interfaceへのhandoff payload
  - Plan Interfaceのどのphaseへ戻すか
  - 修正版Plan/Design LockからExecutionを再生成する規則
  - 古いExecution Stateをsupersedeする規則

  「戻る」というHuman Action名はありますが、実態はExecution終了です。

  ### 2. Blocking Issueの再開操作がWorkflowとAction定義で一致しない

  Plan/Execute workflowにはissue_resolved遷移があります。しかしHuman Action一覧にはissue_resolvedがなく、
  あるのはInterface actionのresume_workflowです。

  加えて、再開時には以下が明確でありません。

  - Blocking Issueを誰がresolvedへ変更するか
  - resume_conditionを誰が再検証するか
  - unresolved Issueを残したままinitial stateへ戻らない保証
  - partialなGit操作後の再開位置

  特にcommit済み・push失敗のケースをexecution_startedから再開すると、二重commitの危険があります。

  ### 3. Release Gate失敗を通常Review findingとして修正できない

  release_gate_failed → fixingですが、apply-scope-fixの入力はlatest_reviewとreview_findingsです。

  Release Gateで初めて発見された問題はReview findingではないため、修正Skillの入力契約と一致しません。例え
  ばfull test failure、secret scan failure、publish scope failureを何に基づいて直すか未定義です。

  最小限、Fix Requestを共通型にして、sourceをimplementation_review | release_gateから選べる必要がありま
  す。

  ### 4. Git Operations AgentのguardsがState遷移へ結合されていない

  workflow.yamlにguardsは定義されていますが、input_validation stateや各transitionからguard名が参照されてい
  ません。.codex/agents/git-operations/workflow.yaml:36

  人間向け文書では前提条件がありますが、machine-readable workflow上では「valid」の判定条件が宣言的に閉じて
  いません。

  ### 5. Release Gateの鮮度保証が弱い

  Git Operations Agentはrelease_gate.checked_revisionを要求しますが、次との一致条件がありません。

  - current HEAD
  - working tree diff fingerprint
  - approved changed files
  - Design Lock revision
  - Review対象implementation revision

  Gate通過後に差分が変わっても、status: passedだけで公開へ進める余地があります。stage直前にGate対象diffと
  の同一性を検証する必要があります。

  ### 6. stage前Release GateとTemplateのStaged Filesが矛盾する

  harness/development/templates/RELEASE_GATE.md:89

  ここはPublish Candidate FilesまたはApproved Changed Filesにすべきです。Gate前stageを許す設計にすると、
  Release Gate前の公開対象操作との境界が曖昧になります。

  ### 7. Design Lock TemplateとSchemaが一致しない

  TemplateのExecution Contractにはworking_branchがありますが、Design Lock Schemaにはありません。

  - .codex/harness/development/templates/DESIGN_LOCK.md:14
  - .codex/harness/development/schemas/design-lock.schema.yaml:55

  Executeのrequired inputではworking branchが必要なため、どの成果物を正本にするか不明になります。

  ### 8. 独立Reviewの実行上の独立性が弱い

  Reviewer Skillは実装を変更しないため、責務上の分離は良好です。しかし、以下は規定されていません。

  - ImplementerとReviewerのcontext分離
  - Reviewerへ実装者の自己評価を渡さない規則

  - 同一AgentがReview結果を書き換えない保証
  - Review artifactのimplementation revision/diff hash固定

  「別Skill」であることは確認できますが、「独立評価」の再現性までは保証できません。

  ## 良い点

  ### AGENTS.mdは概ね薄いルーターとして成立している

  AGENTS.mdはConcept docs、Harness、禁止領域、Git Operations Agentへの入口を示し、詳細WorkflowをHarness側
  へ委譲しています。docs/ja/の禁止境界も明確です。

  ただしGit操作について、Development Harnessの説明とGit Operations Agentルートが実装上競合しているため、
  ルーターとしての最終的な一意性は未達です。

  ### Plan ReviewとImplementation Reviewは概念上分離されている

  - Plan Review: 前提、Decision、Scope、境界、未決事項のレビュー
  - Implementation Review: 実差分、Design Lock、Acceptance Criteria、検証結果のレビュー

  両者の目的と入力は明確に異なり、混同されていません。

  ### Human Submit原則が強く明記されている

  Decision、Plan、Designについて、AI recommendationやoption selectionだけでは確定しない設計です。Decision
  については専用submit-decision Skillもあり、よく分離されています。

  問題は原則ではなく、Plan/ADR/Design成果物へ反映する実行経路の欠落です。

  ### Reviewerが修正しない構造は成立している

  ReviewImplementationはRepositoryとStateを変更せず、修正をApplyScopeFixへ分離しています。.codex/harness/
  development/skills/execute/review-implementation/SKILL.md:19

  これは重要な安全性です。

  ### 修正回数の所有者と最大回数は明確

  - 最大5回
  - Execute Interfaceが管理
  - Reviewerはcounterを所有しない

  この境界はSchema、Interface、workflow、Skillでほぼ一貫しています。

  ### Release GateとGit安全規則は強い

  以下は適切です。

  - Gate passed前のpublish禁止
  - broad staging禁止
  - explicit path staging
  - base branch direct push禁止
  - force push禁止
  - merge/close/branch delete禁止
  - auth login自動実行禁止
  - secret/privacy/raw source scanのfail closed

  .codex/rules/workflow.rules:1も防御層として有効です。

  ## 結論

  現状のHarnessは、次のWorkflowをAIで安全かつ再現可能に最後まで自律・半自律実行できません。

  Plan
  → Plan Review
  → Implementation Design / Design Lock
  → Implementation
  → Implementation Review
  → Fix / Re-review Loop
  → Release Gate
  → Git Operations
  → Pull Request

  思想と個別Skillの責務設計は良好ですが、オーケストレーション契約が閉じていません。厳密にfail closedで動か
  すAIほど、Plan submit、成果物保存、State Patch検証、Git handoffのいずれかで停止します。

  ## 修正優先順位と最小修正案

  1. 公開操作の所有者をGit Operations Agentへ一本化する
      - Execute配下のpublish/PR Skillを削除またはhandoff Skillへ置換
      - release_gate_passed → git_operations_handoff
      - Agent結果だけをExecute Stateへ取り込む

  2. execution-state Schemaと全Skill Patch pathを完全一致させる
      - path名を一つに統一
      - Skill別patch allowlistを自動検証
      - workflow state名もSchema enumと一致させる

  3. Artifact Writer責務をInterfaceへ追加する
      - Skillのartifact_write_requestを検証
      - Template/Schema検証後にatomic write
      - artifact revision/hashをStateへ保存
      - Human Submit時だけaccepted/lockedへ更新

  4. Plan/ADR/DesignのHuman Gate処理を明示する
      - submit-plan SkillまたはInterface action追加
      - submit_adr / defer_adr追加
      - event名をplan_submitted等に統一
      - draftを人間が確認してからsubmitできる二段階処理にする

  5. Design Lock逸脱からPlanへ戻るhandoffを追加する
      - Lock invalidation
      - affected revision
      - blocking evidence
      - return phase
      - revised Design Lock後の新Execution State生成

  6. Fix RequestをReview/Gate共通型にする
      - finding sourceを記録
      - Gate failureからも安全に修正可能にする
      - 修正後は必ずImplementation Reviewを再実行

  7. Git handoffに鮮度・冪等性条件を追加する
      - reviewed diff hash
      - Gate checked diff hash
      - staged diff hash
      - commit SHA
      - remote HEAD
      - 操作journalと再開位置

  この7点を直せば、「条件付きで実運用可能」まで引き上げられます。ファイル変更は行っていません。
