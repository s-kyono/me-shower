  ## AtomicityとConsistency

  Development Harness Artifact向けのatomic write処理は存在しません。

  Source ingestionにはatomic persistence実装とテストがありますが、これはSource Safety Boundary専用であり、
  Development Harness Artifact Writerとして参照・所有されていません。

  現行契約には以下がありません。

  - temp file
  - flush/fsync
  - 再読込
  - pre/post hash
  - atomic rename
  - CAS
  - file lock
  - Stateとのtransaction
  - orphan検出
  - temp cleanup

  ### 現行の更新順序

  更新順序自体が未定義です。

  ### 推奨する最小整合性モデル

  ローカルFS前提では、次が最小です。

  1. State revisionと既存Artifact hashをCAS検証
  2. Path Policyとsecurityを検証
  3. temp領域へCandidate保存
  4. 再読込、Schema、hash検証
  5. revision付きArtifactをatomic renameで確定
  6. Artifact Referenceを生成
  7. StateをCAS更新
  8. State更新失敗時はArtifactをorphan候補として残す

  Stateを先に更新する方式は、存在しないArtifact Referenceを作るため避けるべきです。

  Artifact先行ではorphanが発生し得ますが、dangling State Referenceより安全です。ただし、次が必要です。

  - request ID
  - Artifact type
  - subject revision
  - content hash
  - orphan識別情報
  - 再実行時の同一性判定
  - 後続回収ルール

  完全なtransaction manifestは初回実装の必須とは限りませんが、少なくともwrite resultまたはpending request
  を復旧可能な形で識別する必要があります。

  - passedはSnapshot/revision/scan Guardで制約される
  - Artifact本文の保存とGate State更新は結び付いていない

  Writerは保存だけを行い、accepted、locked、canonicalを決定してはいけません。CanonicalityはHuman Actionま
  たはWorkflow Guardによって別に決める必要があります。

  ## IdempotencyとConcurrency

  現在は次の契約がありません。

  - Write Request ID
  - idempotency key
  - expected hash/revision
  - create-if-absent
  - replace-if-hash-matches
  - Artifact lock
  - State revisionとArtifact writeのCAS
  - 同一revisionへの異なるcontent拒否
  - crash後temp file処理
  - timeout後lock回収
  - orphaned request検出

  ホストマシン上でも、Agent/Subagentや再実行が重なるとlast-writer-winsになる可能性があります。

  Artifact Persistence単体の最小要件は以下です。

  - Artifact logical key単位のlock
  - source State revision
  - expected previous Artifact hash
  - target Artifact revision
  - content hash
  - request ID/idempotency key
  - 同一request再実行時は同じResultを返す
  - 同一revision・異なるhashはblocked
  - symlinkを追跡してroot外へ出ないことを確認
  - stale lockの自動回収は後続でもよいが、破棄はHuman確認可能にする
