# /brushup:Execute InterfaceとGit Operations Agentの責務が競合している

## 残る軽微な指摘

  責務競合の再発要因ではありませんが、次は後続改善を推奨します。

  1. Execute workflowのimmediate_block_conditionsにgit_authentication_failedが残っています。Executeは
     GitHub認証確認を行わないため、Repository Publish Agent側だけに置く方が境界が明確です。

  2. canonical snapshot hashへ現在のHEAD SHAを含めることを推奨します。現在の定義はpath、Git status、
     content hashが中心なので、HEADだけが変化したケースも確実に検出できるようになります。
      - .codex/agents/repository-publish/shared/repository-publish-policy.yaml:9

  3. canonical snapshot生成は文書契約のみです。実装差によるhash不一致を防ぐため、将来的に共通スクリプトと
     正常系・改変検出テストを追加すべきです。

## 残存リスク

- 上位Agent Routerは外部実行主体であり、実環境とのE2E検証が必要です。
- atomic claim後にAgentが異常終了するとrunning claimが残ります。安全側に停止しますが、自動回収は未実装です。

- commit成功後・Result保存前の異常終了について、完全なoperation journalはありません。
- 全体テスト4件はprivate profile data不在のため失敗しています。

## 次に修正すべき総合テスト指摘

1. 上位Agent RouterからInvocation Requestを解釈するE2Eテスト
2. stale run claimの検出・Human Review付き回収
3. commit成功直後のAgent crashを想定した冪等性テスト
4. symlinkとsubmoduleの専用Snapshot fixture
5. private dataなしでも全体テストを再現可能にするsynthetic fixture境界の整備

結論として、今回の修正によりRelease Gate後のRepository Publishは、リポジトリ内の契約・検証・Snapshot・重
複防止の範囲では安全かつ再現可能です。実運用開始条件は、上位Agent Routerが構造化Invocation Requestとrun
registry契約を遵守することです。
